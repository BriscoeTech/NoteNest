# NoteNest Development Process Guide

This file contains operational runbooks and command workflows.
Architecture requirements belong in `Architecture.md`.

## 0. Documentation File Conventions

- Markdown guide files must be stored in the project root directory.
- Markdown guide filenames must use PascalCase (for example: `Architecture.md`, `SessionChangeLog.md`, `DevelopmentProcessGuide.md`).

## 0.1 Markdown Guide Purposes

- `Architecture.md`
- What goes here: product requirements, feature contracts, data contracts/invariants, non-functional requirements, and architecture rationale/tradeoffs.
- Why: this is the canonical specification used to understand or rebuild the system correctly.

- `DevelopmentProcessGuide.md`
- What goes here: operational runbooks (build/release commands, local dev server workflows), and documentation conventions.
- Why: keeps execution procedures in one place so work is consistent and repeatable.

- `SessionChangeLog.md`
- What goes here: chronological session-level change history (what changed, when, and brief implementation notes).
- Why: preserves historical context without polluting the architecture specification.

- `README.md`
- What goes here: project overview, core capabilities, and quick-start usage/build/deploy basics for contributors and users.
- Why: provides the first entry point for understanding and running the project.

## 1. Build Identity Workflow

### Policy
- The app does not maintain a manual application version.
- `build-info.json` is the runtime source of build identity.
- `build-info.json` is generated by the build pipeline and must not be edited manually.
- Runtime build identity is the `builtAt` timestamp.
- Other version fields (for example in `package.json`) are package metadata only and must not be used for app runtime/build identity.
- Sidebar footer display must show a human-readable last-build day/date/time derived from `builtAt`.

### Convention and Rationale
- Build timestamps avoid stale manual version numbers and identify the exact deployed build.
- Runtime build-info loading from `build-info.json` keeps static deploys inspectable after rebuild and push.
- Service-worker/cache identity must derive from `builtAt` so each production build gets a fresh cache.
- UI and export metadata should use the live runtime `builtAt` from `build-info.json` when available, and may fall back to the last successfully loaded cached runtime `builtAt` if live retrieval fails.
- If neither live nor cached runtime build info is available, UI/export may degrade to an explicit unknown value.
- Avoid defining app identity in multiple places to prevent drift and stale runtime behavior.

### Commands
- Production rebuild:
```bash
npm run build
```

### Implementation Rule
- Build scripts must emit `build-info.json` with a `builtAt` timestamp.
- `package.json` version must not be used as product/app identity.
- App identity-related code must not read `package.json` for runtime or build-time identity.

### Windows PowerShell Note
- If `npm` script execution is blocked by policy, use:
```powershell
npm.cmd run build
```

## 2. Build and Release Artifacts

- Production build command:
```bash
npm run build
```
- Build script: `script/build.ts`
- Output location: `docs/`
- `docs/` is a required tracked deploy artifact for GitHub Pages.

### 2.1 Runtime Refresh Verification
- Validate both refresh paths after service-worker related changes:
- Normal browser refresh must render app shell correctly.
- Sidebar `Refresh` must reload without clearing service workers/cache.
- Offline startup must still load the cached app shell with no network available.

### 2.2 Line Ending Policy
- `.gitattributes` is the repository source of truth for line-ending behavior.
- This project is edited and synced across Linux and Windows machines using syncing software.
- Source files, config files, docs, JSON, CSS, HTML, and shell scripts must use `LF`.
- Windows command files (`.bat`, `.cmd`) must use `CRLF`.
- Do not remove or weaken `.gitattributes` rules unless replacing them with an equally explicit cross-platform policy.
- If line-ending noise reappears, prefer fixing repo policy and normalization rather than committing mixed-EOL churn.

## 3. Local Dev Server Workflow

### Required Scripts
- Always use vetted scripts for start/stop to reduce environment and process-management errors.
- Available scripts:
- `script/dev-server-start.sh`
- `script/dev-server-stop.sh`
- `script/dev-server-start.bat`
- `script/dev-server-stop.bat`

### Windows Usage
- Start:
```powershell
script\dev-server-start.bat
```
- Stop:
```powershell
script\dev-server-stop.bat
```

### Linux Usage
- Start:
```bash
./script/dev-server-start.sh
```
- Stop:
```bash
./script/dev-server-stop.sh
```

## 4. Recommended Pre-Commit Checks

- Type check:
```bash
npm run check
```
- Rebuild docs when releasing:
```bash
npm run build
```

## 5. Release Verification Checklist

- `npm run build` generated `docs/build-info.json`.
- Last build timestamp appears correctly in the sidebar footer.
- Build identity consumers (export metadata, cache-busting, service worker cache identity) are reading from `build-info.json`.
- Type check passes:
```bash
npm run check
```
- Production build succeeds and generates deployable `docs/` artifacts:
```bash
npm run build
```
